## ワークフロー
- 3ステップ以上のタスクは必ずPlanモードで計画してから実行
- タスク完了前にテスト・動作確認を必ず行う
- 問題が起きたらすぐSTOPして再計画する

## コーディング原則
- シンプルさ優先、最小限の変更のみ
- 一時的な修正はしない、根本原因を探す
- 変更は影響範囲を最小限に抑える

## 自己改善ループ
- ユーザーに修正されたら tasks/lessons.md にパターンを記録
- セッション開始時に tasks/lessons.md をレビューする
- 同じミスを繰り返さない

## タスク管理
- 計画は tasks/todo.md に書く
- 完了したらチェックを入れる
- 変更のたびに高レベルサマリーを説明する

## 解説書 (teach.md)
- プロジェクトルートに `teach.md` を維持する（学習用解説書）
- detail.md は「**何があるか**」のリファレンス、teach.md は「**なぜ・どう動くか**」の教科書
- 対象は初学者（プログラミング初めて）。チュートリアル + コラム形式
- 新機能/重要パターンが追加されたとき、teach.md にも対応する解説章を追加するか検討
- 細かい変更で teach.md を毎回更新する必要はない（detail.md / update.md と役割を分ける）

## ファイル構成ドキュメント (detail.md)
- プロジェクトルートに `detail.md` を維持する
- **新規ファイルを作成したら必ず detail.md に追記する**（同じターン内で）
- **既存ファイルを大幅変更（新メソッド追加・責務変更・依存先変更）した場合も detail.md を更新する**
- ファイル削除・リネーム時も detail.md を更新する
- detail.md の記載項目（各ファイルごと）:
  1. ファイルパス
  2. 役割（1〜2行）
  3. 主要な型・メソッド一覧と短い説明
  4. 依存しているファイル（import/参照）
  5. このファイルを参照しているファイル
- 更新を忘れないため、ファイル作成・編集タスクの最後に必ず detail.md を確認する

## 進捗ログ (update.md)
- プロジェクトルートに `update.md` を維持する（追記型・新しいものが上）
- **ユーザーから新しい作成・変更依頼を受けたとき**: その場で「立案」エントリを追記する
  - 何を依頼されたか
  - どう実装するつもりか(計画の要約)
  - 影響範囲(どのファイル・どの機能)
- **作業を進めたとき**: 同じエントリに「進捗」を追記する
  - 完了した内容
  - 残課題
  - ブロッカーがあれば明記
- **作業完了時**: 「完了」マークを付ける
- エントリ形式: `## YYYY-MM-DD HH:MM — タイトル` で日付見出し、本文に立案/進捗/完了を記載
- 立案なしで実装を始めない。必ず先に update.md に書いてからコードに触る

## ブランチ運用
- **新機能の実装・追加作業は必ず新規ブランチで行う**
- ブランチ名形式: `feature/YYYYMMDD-HHMM-<short-name>` (例: `feature/20260518-0410-meal-photo`)
  - 日時は作成時点の現在時刻
  - `<short-name>` は機能を表すケバブケース3〜5語
- main に直接コミットしない(ドキュメントだけの軽微な更新は許容するが、念のため確認)
- ブランチを切る前に `git status` で作業ツリーがクリーンか確認
- マージはユーザーが PR レビュー後に行う。Claude は merge/push を勝手にしない
- ブランチ名生成時は `date +%Y%m%d-%H%M` を使って正確な現在時刻を取得

## 引き継ぎノート (baton.md)
- トークン残量が**逼迫してきた**ら、次セッションへの引き継ぎを `baton.md` に書く
- タイミングの目安:
  - コンテキスト圧縮の予兆を感じたら(長文出力後・大量サブエージェント呼び出し後)
  - フェーズの切れ目に到達したとき
  - ユーザーから「baton書いて」と言われたとき
- 書く内容(テンプレ):
  1. **現在地**: どのフェーズ/タスクのどこまで完了したか
  2. **進行中の作業**: 今やっている途中のこと(ファイル名・状態)
  3. **次にやること**: 具体的な次ステップ(コマンド/ファイル/agentの指示まで含めて)
  4. **重要な決定事項**: このセッションで決まった設計判断
  5. **ハマりポイント**: 注意すべき罠
  6. **参照すべきドキュメント**: update.md / detail.md / todo.md の該当箇所
- baton.md は**上書き型**(常に最新の引き継ぎ1つだけ)
- 書いたら短く「baton.md 更新済み。次セッションへの引き継ぎ準備完了」とユーザーに伝える
