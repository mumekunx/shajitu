# teach.md — 教育用サイバーセキュリティ通信シミュレーター 解説書

この文書は、プログラミングを始めたばかりの人がこのプロジェクトのコードを読んで「なぜこう書いてあるのか」「どう動いているのか」を理解できるようにするための教科書です。

`detail.md` が「このファイルには何がある」を網羅するリファレンスだとすれば、この `teach.md` は「なぜそう作ったのか」「動かすとどんな順序で何が起きるのか」を、ストーリーとして追う読み物です。すべてのファイル・関数を説明するわけではないので、網羅的なリストが欲しいときは `detail.md` を見てください。

## 第1章: このプロジェクトは何か

このプロジェクトは「ゲーム」ではありません。スコアを競ったり、敵を倒したりする目的のソフトウェアではなく、**ネットワーク上でサイバー攻撃がどう起きるかを目で見て理解するための可視化ツール**です。

画面の左側にはネットワーク図(インターネット・ファイアウォール・ルーター・サーバー・PCなど)が表示され、通信は線ではなく光る粒子としてノード間を流れます。攻撃ボタンを押すと、攻撃者ノードから標的ノードに向かって赤い粒子が流れ、ファイアウォールがそれを弾いたり、稀に侵入を許してノードが「感染」したりする様子が見えます。右側にはイベントログ、下にはタイムライン、右下には統計パネル(CPU/Memory/Network/AttackScore)が表示されます。

つまりこのツールの目的は、「SSHブルートフォース攻撃とは何か」「DDoSはどういう見た目の現象か」を、文章で読むのではなく、動くアニメーションとして体験してもらうことです。教育目的であるということは、コードの正確さよりも「理解しやすさ」「見た目のわかりやすさ」を優先する設計判断がいくつもある、という意味でもあります(例えば、攻撃の成功確率は実際のセキュリティ統計ではなく、デモとして適度に盛り上がるように調整された数値です)。

## 第2章: なぜReact + TypeScript + Viteなのか

この3つはそれぞれ役割が違う道具です。初めてこの組み合わせを見ると「なぜ3つも必要なのか」と思うかもしれませんが、実際にはレイヤーが分かれています。

- **React**: 「画面に何を表示するか」を、データ(状態)に応じて自動的に組み立て直してくれるライブラリです。たとえば「攻撃中のノードは赤く光らせる」というルールを一度書いておけば、データが「攻撃中」に変わった瞬間にReactが自動的にその部分だけ再描画してくれます。自分で「ここを赤くして、あそこを元に戻して」と手続き的にDOMを書き換える必要がありません。
- **TypeScript**: JavaScriptに「型」という安全装置を追加した言語です。例えば `NetworkNode` というデータには `id` や `health`(0〜100の数値)といったプロパティがあると決めておけば、存在しないプロパティを参照したり、文字列を入れるべき場所に数値を入れたりすると、実行前(コードを書いている最中)にエラーとして教えてくれます。複数人(このプロジェクトでは複数のサブエージェント)が同時に別々のファイルを書くときに、「このデータの形はこうだ」という契約をコードレベルで強制できるのが大きな利点です。実際に `src/types/index.ts` がその「契約書」の役割を果たしています。
- **Vite**: 開発中にコードを保存した瞬間、ブラウザに即座に反映してくれる開発サーバー兼ビルドツールです。古い世代のビルドツールは変更ごとに数秒〜数十秒待たされることがありましたが、Viteは基本的に一瞬で反映されます。最終的に公開する際は、TypeScriptのコードをブラウザが読めるJavaScriptにまとめて出力する役割も持ちます(`npm run build`)。

まとめると、TypeScriptが「正しいデータの形」を保証し、Reactが「データに応じた画面の自動更新」を担い、Viteが「開発と本番ビルドの土台」を提供する、という3層構造です。

## 第3章: 「状態管理」の考え方

### Reactの`useState`だけでは足りない理由

Reactには `useState` という、コンポーネント内で「変化する値」を持つための基本機能があります。小さなアプリならこれで十分です。しかしこのプロジェクトには次のような事情があります。

- ノードの状態(感染しているか、体力はいくつか)を、`NetworkMap` だけでなく `EventLog` や `Timeline` など複数のコンポーネントが参照する必要がある。
- 攻撃の進行ロジック(`src/engine/`)は、Reactコンポーネントの外、つまりタイマーや`requestAnimationFrame`のループの中で動く。Reactのコンポーネントの外からは `useState` は使えません。

`useState` は「そのコンポーネントの中だけの記憶」です。複数の場所(画面の複数箇所、かつReactコンポーネントの外側のロジック)から同じデータを読んだり書いたりしたい場合、`useState` をバケツリレーのように親から子へ延々と渡していく(「プロップスのバケツリレー」と呼ばれます)のは非常に面倒で、ミスも増えます。

### zustandストアとは何か

`zustand` は、コンポーネントの外側に「みんなが読み書きできる共有の保管庫(ストア)」を1つ作るための仕組みです。`src/store/simulationStore.ts` がまさにそのストアの実体です。実際の構造を見てみましょう。

```ts
interface SimulationStore {
  nodes: NetworkNode[];        // ネットワーク上の全ノード
  links: NetworkLink[];        // ノード間の全リンク(通信路)
  logs: LogEntry[];            // イベントログの履歴
  timeline: TimelineEvent[];   // タイムライン上のイベント履歴
  stats: SimStats;             // CPU/Memory/Network/AttackScore
  activeAttacks: AttackState[]; // 現在進行中の攻撃一覧

  updateNode: (id, patch) => void;
  updateLink: (id, patch) => void;
  addLog: (entry) => void;
  // ...ほか
}
```

このプロジェクトで「画面に表示される、変化しうるすべてのデータ」(ノード、リンク、ログ、タイムライン、統計、進行中の攻撃)が、この1つのストアに一元管理されています。

ストアを使う側のコード(例えば `NetworkMap.tsx`)は `useSimulationStore()` を呼ぶだけで最新の `nodes` や `links` を取得でき、ストアの値が変わると自動的に再レンダリングされます。一方、Reactコンポーネントの外側にある `src/engine/simulationEngine.ts` のようなコードは、`useSimulationStore.getState()` という形でフック(`use~`)を使わずに直接ストアの現在値を読んだり、`updateNode` などの関数を呼んで更新したりできます。これが「Reactの外からも読み書きできる共有の保管庫」という性質の具体的な使われ方です。

`updateNode` の実装を見ると、考え方がよくわかります。

```ts
updateNode: (id, patch) =>
  set({
    nodes: get().nodes.map((node) => (node.id === id ? { ...node, ...patch } : node)),
  }),
```

これは「`id`が一致するノードだけ、指定された差分(`patch`)で上書きした新しい配列を作って、ストア全体を入れ替える」という処理です。既存のノードを直接書き換えるのではなく、常に「新しい配列・新しいオブジェクト」を作って差し替えているのがポイントです(これを「イミュータブル(不変)な更新」と呼びます)。Reactは「前の値と新しい値が違うオブジェクトかどうか」を見て再描画するかを判断する仕組みを使っているため、この書き方が前提になっています。

## 第4章: パフォーマンスの肝

ここがこのプロジェクトの設計で最も重要な部分です。

### なぜノード(11個)はzustand+Reactで、粒子(数百個)は別の仕組みなのか

このシミュレーターには2種類の「動くもの」があります。

1. **ノード**(インターネット・ファイアウォール・サーバーなど): 全部で11個。状態は「感染しているか」「体力はいくつか」など、頻繁には変わりません。
2. **粒子**(ノード間を流れる光の点): 攻撃中は同時に数百個存在し、すべてが毎フレーム(1秒に60回)位置を変えます。

11個程度のノードであれば、zustandストアに状態を持たせて、変化があったときにReactに再レンダリングさせても全く問題ありません。Reactの「差分を見て必要な部分だけ更新する」という仕組みは、対象が少数であればとても効率的に働きます。

しかし、数百個の粒子を同じやり方で扱うと話が変わってきます。粒子は1フレームごとに位置(`t`、リンク上の進行度)が変わります。これをzustandストアに乗せて、毎フレーム「数百個分のオブジェクトを新しく作り直してストアにセットする」とどうなるか考えてみましょう。

- ストアが変化した、とReactに伝わるたびに、Reactは「画面のどこを再描画すべきか」を計算するための処理(差分検出・再レンダリングの判定)を行います。この処理自体に時間がかかります。
- 11個のノードなら一瞬で終わるこの判定が、数百個のオブジェクトを毎フレーム(60分の1秒ごと)処理するとなると、ブラウザが追いつかなくなり、アニメーションが滑らかに動かなくなります(いわゆる「フレームが落ちる」「コマ落ちする」状態)。

つまり、Reactは「データが変わったらどこを直すか自動的に考えてくれる」便利な仕組みですが、その「自動的に考える」という処理自体にコストがあり、更新頻度・更新対象の数が増えるとそのコストが支配的になってしまうのです。

### Canvas 2Dという別解

そこで粒子の描画には、Reactを経由しない別の仕組みが使われています。`src/engine/particleSystem.ts` を見てみましょう。

```ts
class ParticleSystem {
  private particles: Particle[] = [];

  spawnForLink(link: NetworkLink, count: number): void { /* 粒子を追加 */ }
  tick(dtMs: number): void { /* 全粒子の位置(t)を進め、ゴールに着いた粒子を消す */ }
  getParticles(): readonly Particle[] { return this.particles; }
}

export const particleSystem = new ParticleSystem();
```

これは普通のJavaScriptのクラスのインスタンス(シングルトン)で、zustandストアにも、Reactのコンポーネントツリーにも属していません。粒子の配列はこのオブジェクトが内部に直接持っているだけです。

描画側(`src/components/NetworkMap/ParticleCanvas.tsx`)は、HTML標準の`<canvas>`要素に対して、`requestAnimationFrame`で組んだ自前のループから直接「ここに丸を描いて」という命令を出します。これがCanvas 2Dという描画方式です。Reactの「データが変わったらどこを直すか考える」という仲介processがなく、毎フレーム画面全体を一度クリアして、全粒子をその時点の位置に応じてそのまま描き直すだけです。「絵を直接描く」だけなので、Reactのレンダリング判定のオーバーヘッドが一切発生しません。1500個までの粒子(`MAX_PARTICLES`)を許容できるのも、この仕組みのおかげです。

### D3.jsとCanvas描画の役割分担

もう一つの分業が、ノードの「座標計算」と「見た目の描画」の分離です。

`src/components/NetworkMap/useNetworkLayout.ts` が使っている `d3-force` は、「ノードどうしがバネで繋がっていて、互いに反発しあいながら自然な位置に収まっていく」というシミュレーション(力学レイアウト)を計算するライブラリです。これは「ノードがどの座標(x, y)にあるべきか」という**数字を計算するだけ**の仕事で、画面に何かを描いたりはしません。

```ts
const sim = forceSimulation<LayoutNode>(layoutNodes)
  .force('link', linkForce)        // リンクで繋がったノードを引き合わせる
  .force('charge', forceManyBody().strength(-260)) // ノード同士を反発させる
  .force('center', forceCenter(width / 2, height / 2)) // 全体を中央に寄せる
  .force('collide', forceCollide(...)) // ノードが重ならないようにする
  .alphaDecay(0.04); // 徐々に落ち着いて止まる
```

計算結果(各ノードの`x`, `y`)は、Reactのローカルな`state`(`positions`)に格納され、それを使って実際の描画(SVGでのノードの円や粒子の経路の基準点)が行われます。重要な工夫として、このフックは「ノードの集合(`nodes.length`)、画面サイズ(`width`/`height`)が変わったときだけ」シミュレーションを再構築するようにコメントで明示されています。レイアウトの計算自体も、力が均衡して落ち着く(`alphaDecay`によって徐々に動きが小さくなる)と自動的に止まるため、永遠に毎フレーム再計算され続けるわけではありません。

まとめると、役割分担はこうなります。

- **D3.js(`useNetworkLayout.ts`)**: ノードが「どこにあるべきか」という座標を計算する(数学の問題を解くだけ)。
- **Canvas 2D(`ParticleCanvas.tsx`)**: 粒子の「見た目」を、計算済みの座標を使って直接描画する(絵を描くだけ)。
- **React + zustand**: ノードや攻撃状態など「個数が少なく、頻繁には変わらないデータ」の管理と、それに応じたSVG部分(ノード本体やエフェクト)の描画。

「計算」「描画」「状態管理」という3つの役割が、それぞれ最も向いている道具に分かれている、というのがこのプロジェクトのパフォーマンス設計の核心です。

## 第5章: 攻撃シミュレーションの仕組み

攻撃が始まってから終わるまでの流れを、最も単純な `runPing`(Ping攻撃)を例に1ステップずつ追ってみます。

### ステップ0: ボタンが押される

`AttackToolbar.tsx` でユーザーがボタンを押すと、`triggerAttack(type, sourceId, targetId)` が呼ばれます。これは `src/engine/simulationEngine.ts` で定義されている関数です。

### ステップ1: `triggerAttack` が攻撃を「登録」する

```ts
export function triggerAttack(type: AttackType, sourceId: string, targetId: string): void {
  const { addAttack, addLog, addTimelineEvent } = useSimulationStore.getState();

  const attackId = addAttack({ type, sourceId, targetId });

  addLog({
    level: 'warning',
    type,
    message: `攻撃を検知: ${type} (${sourceId} -> ${targetId})`,
  });
  addTimelineEvent({ label: `攻撃開始: ${type}`, kind: 'attack_start' });

  runAttackScript(type, attackId, sourceId, targetId);
}
```

まず `addAttack` がストアに新しい `AttackState`(進行中の攻撃を表すデータ)を1件追加し、一意の `attackId` を発行します。次に「攻撃を検知しました」という警告ログとタイムラインイベントを追加します。最後に `runAttackScript` を呼んで、攻撃種別ごとの実際の進行ロジックに処理を渡します。

### ステップ2: `runAttackScript` が振り分ける

`src/engine/attackScripts.ts` の `runAttackScript` は、`switch`文で攻撃種別に応じて `runPing` / `runPortScan` / `runSshBruteforce` などの専用関数を呼び分けるだけのディスパッチャです。

### ステップ3: `runPing` がタイマーでシナリオを進める

```ts
function runPing(attackId: string, sourceId: string, targetId: string): void {
  const { updateAttack, removeAttack, addLog } = useSimulationStore.getState();
  const link = findLink(sourceId, targetId) ?? findAnyLinkInvolving(sourceId);

  const totalDurationMs = 2000;
  const steps = 4;
  const stepMs = totalDurationMs / steps;

  if (link) {
    flashLinkIntensity(link, Math.min(1, link.intensity + 0.3), totalDurationMs);
  }

  for (let i = 0; i < steps; i++) {
    schedule(() => {
      if (link) {
        particleSystem.spawnForLink({ ...link, trafficType: 'attack' }, 2);
      }
      updateAttack(attackId, { progress: (i + 1) / steps });
    }, i * stepMs);
  }

  schedule(() => {
    addLog({ level: 'info', type: 'ping', message: `Ping応答あり (${sourceId} -> ${targetId})` });
    removeAttack(attackId);
  }, totalDurationMs);
}
```

ここで使われている `schedule` という関数は、内部的には `setTimeout` をラップしたヘルパーです。「○ミリ秒後にこの処理を実行する」というタイマーを仕掛けて、後から `stopEngine()` が呼ばれたときに全部まとめて解除できるよう、発行したタイマーIDを `activeTimers` という集合に記録しています。

Ping攻撃全体は2秒間(`totalDurationMs = 2000`)続く設計で、4ステップ(`steps = 4`)に分けて進みます。`for`ループは「0ms後・500ms後・1000ms後・1500ms後」のそれぞれに `schedule` を仕掛けるという意味で、実際にループの中で粒子が即座に動くわけではありません。各ステップが来るたびに、

- リンク上に攻撃トラフィックを表す粒子を2個生成する(`particleSystem.spawnForLink`)
- 攻撃の進行度(`progress`)を `1/4`, `2/4`, `3/4`, `4/4` と更新する

という処理を行います。そして最後、2000ms経過したタイミングでもう一つの `schedule` が実行され、「Ping応答あり」という情報ログを出し、`removeAttack(attackId)` でストアから攻撃の記録を削除します。これにより画面上の演出(進行表示など)もまとめて消えます。

つまり「攻撃が始まって、しばらく粒子が流れて、最後に結果が出て消える」という一連の流れは、複数の `setTimeout` を時間差で積んでおくことで表現されています。他の5攻撃(PortScan/SSHBruteForce/SQLInjection/DDoS/Ransomware)も基本的には同じ骨格(`schedule`の連鎖→`updateAttack`での進行更新→最後に`addLog`/`removeAttack`)で書かれていますが、試行回数や成功・失敗の分岐、ノードの感染処理などがそれぞれ追加されています。

## 第6章: コラム — 実際に見つかったバグから学ぶ

開発の最終段階で、実際にブラウザで全攻撃ボタンをクリックして動作確認(QA)した際、SSHブルートフォース攻撃で次のような奇妙なログが出ることが見つかりました。

```
攻撃を検知: ssh_bruteforce (attacker-1 -> host-1)
...
SSHブルートフォース攻撃が成功しました (server-web)
```

「`host-1` への攻撃」と表示されたのに、成功した瞬間だけ別のノード(`server-web`)の名前が出てくる、という矛盾です。

原因を簡略化した擬似コードで示すと、次のような状況でした。

```ts
function runSshBruteforce(attackId, sourceId, targetId) {
  // targetId は「攻撃の本当の標的」を表す、関数の引数として渡された値
  // (例: targetId === "host-1")

  // ところが、リンク(通信路)の一覧から「ssh通信のリンク」を検索していた
  const link = findLinkByTrafficType('ssh'); // 偶然見つかったリンクが server-web 絡みだったとする

  if (攻撃成功) {
    // ここで本来は targetId ("host-1") を使うべきところを、
    // link から逆算したノードID ("server-web") を使ってしまっていた
    updateNode(link.target, { infected: true }); // 誤り
    addLog({ message: `成功しました (${link.target})` }); // 誤り
  }
}
```

`targetId`(関数の引数として明示的に渡された、本当に正しい値)と、`findLinkByTrafficType` で見つけたリンクから逆算したノードID(「たまたまその種類の通信を行っているリンク」を検索した結果)は、**多くの場合は同じ値になりますが、必ず同じになる保証はありません**。トポロジ上に複数のSSHリンクがあったり、検索条件に複数のリンクが当てはまったりすると、「検知時に表示した標的」と「成功時に処理した標的」がズレてしまいます。

実際の修正後のコードでは、`updateNode`や`addLog`に渡す値として、`link`から逆算した値ではなく、関数の引数である `targetId` をそのまま使うように直しました。

```ts
updateNode(targetId, { infected: true, health: 15 }); // targetId(引数)を直接使う
addLog({ message: `SSHブルートフォース攻撃が成功しました (${targetId})` });
```

ここから学べる教訓は、「**同じはずの値を、複数の異なる経路(関数の引数として渡された値 / データ構造を検索して逆算した値)から求めると、データの組み合わせ次第で食い違うことがある**」という点です。今回のように「複数人(または複数のサブエージェント)が並列で似たようなコードを書く」場面では、片方が「引数をそのまま使う」、もう片方が「データから再計算する」という別々のアプローチを取ってしまい、見た目は同じロジックのつもりでも結果がズレる、という事故が起きやすくなります。

対策としては、「すでに正しい値が手元にある(関数の引数など)なら、それを使い、わざわざ別の場所から再計算しない」というのが基本です。もし2つの経路で求めた値が一致するはずだと考えているなら、コードレビューの際に「この値、どこから来ている?」を辿って確認する習慣が有効です。

## 第7章: 今後拡張するなら

新しい攻撃タイプ(例: `phishing` のようなもの)を追加したい場合、おおまかに以下のファイルに手を入れることになります。

1. **`src/types/index.ts`**: `AttackType` という型(現在は `'ping' | 'port_scan' | 'ssh_bruteforce' | 'sql_injection' | 'ddos' | 'ransomware'` の6種)に新しい文字列を追加します。これが全ての「契約」の出発点です。
2. **`src/engine/attackScripts.ts`**: `runPing` や `runSshBruteforce` のような専用関数(`runXxx`)を新しく書きます。`schedule` を使ったタイマー管理、`updateAttack`での進行度更新、最後の`addLog`/`removeAttack`という骨格を参考にします。書いた関数は、ファイル末尾の `runAttackScript` の `switch` 文に新しい `case` を追加して呼び出されるようにします。
3. **`src/components/NetworkMap/AttackToolbar.tsx`**: `ATTACK_TYPES` の配列に新しい種別を追加し、必要なら `TARGET_TYPE_PRIORITY` にどの種類のノードを標的にしやすいかを設定します。これでツールバーに新しいボタンが自動的に表示されます。
4. **`src/components/NetworkMap/AttackEffectsLayer.tsx`**(必要な場合): その攻撃に専用の見た目の演出(リング、点滅、テキストなど)を追加したい場合はここに分岐を加えます。何もしなければ既存の汎用的な見た目になります。
5. **`src/components/NetworkMap/constants.ts`**: 新しい攻撃のラベル(`ATTACK_LABELS`)や演出色(`ATTACK_COLORS`)もここに追加しておくと、ツールバーや演出が正しい表示になります。

このように「型定義 → 進行ロジック → UIボタン → 専用ビジュアル(任意) → 色・ラベル定数」という順番で1つずつファイルを辿っていけば、既存の6攻撃と同じ構造の新しい攻撃を追加できます。

## 第8章: 教育的リアリティを高める4つの工夫

第1章で触れたとおり、このツールは「正確さより理解しやすさ」を優先する場面がいくつもあります。しかしそれは「現実を無視していい」という意味ではありません。この章では、デモとしての単純さを保ったまま現実との接点を増やすために加えた4つの小さな工夫を、「なぜそうしたか」を中心に見ていきます。

### 8-1: 実在ツール名の提示 — ツールチップという最小限のUIで補う

`AttackToolbar.tsx`の各攻撃ボタンには、`title={`実在ツール: ${ATTACK_TOOLS[type]}`}`という形でネイティブのツールチップ(ボタンにマウスを乗せたときに出る吹き出し)が付いています。中身は`src/components/NetworkMap/constants.ts`の`ATTACK_TOOLS`という単純な`Record<AttackType, string>`で、例えば`ssh_bruteforce`には`'Hydra / Medusa'`、`sql_injection`には`'sqlmap'`という、実際にセキュリティの現場で使われるツール名が対応しています。

ここでの設計判断のポイントは、「新しいUI部品を作らない」ことです。バッジや別パネルを追加することもできましたが、それは画面の情報量を増やし、シミュレーターの主役である「粒子が流れる様子」から注意を逸らしてしまいます。ブラウザ標準の`title`属性は、追加のコンポーネントもCSSもほぼ不要で、「知りたい人だけがホバーして知る」という形で情報を差し込めます。教育コンテンツにおいて「主役の邪魔をしない補足情報」をどう出すかという問題に対する、最小コストの答えの一つです。

### 8-2: 豆知識(trivia)ログ — 単純化を注釈で正直に補う

このシミュレーターの攻撃成功率(例えばSQLインジェクションの50%)は、現実の統計ではなく「デモとして適度に盛り上がる」ように調整された数値だと第1章で説明しました。これは教育ツールとして避けられない単純化ですが、それを黙って提示すると「現実もこんなものだ」という誤解を生みかねません。

そこで、各攻撃が終了する瞬間に、`level: 'trivia'`という専用のログレベルで「現実ではこう」という一文を追加しました。例えば`runSshBruteforce`の終了時には、実際のブルートフォース攻撃が数万〜数百万回の試行を要すること(このデモではたった10回で決着している)への注釈が入ります。派手な`critical`(赤)や`success`とは違う、落ち着いたindigo系の配色(`getLevelStyle`の`trivia`ケース)にしているのも、「警告」ではなく「補足知識」であることを色で伝えるためです。

実装として面白いのは、これが`LogLevel`という型への**加算的拡張**の実例になっている点です。`'info' | 'warning' | 'critical' | 'success'`という既存のunion型に`'trivia'`を1つ追加しただけですが、TypeScriptの`strict`設定のもとでは、この型を`switch`文などで網羅的に扱っている箇所(`logStyles.ts`の`LEVEL_STYLES`)で「新しいケースの処理漏れ」がすぐ型エラーとして表面化します。契約(型)を1箇所広げるだけで、その契約に依存する全箇所が「対応済みか」を自動でチェックしてくれる、TypeScriptらしい安全性の使い方です。

### 8-3: SSH認証方式による成功率の可変化 — 「対策が効く」ことを数字の変化で体験する

`AttackToolbar.tsx`には「SSH認証:」というラベルの付いたトグル(パスワード認証/公開鍵認証)があります。これを切り替えてからSSHブルートフォース攻撃を実行すると、`runSshBruteforce`内の成功確率が`authMethod === 'publickey' ? 0.005 : 0.15`という形で変わります。パスワード認証なら15%、公開鍵認証ならほぼ0.5%まで下がります。

なぜ公開鍵認証だとほぼ成功しなくなるのか。パスワード認証は「正しい文字列を言い当てれば突破できる」ため、辞書攻撃やブルートフォース(総当たり)が原理的に有効です。一方、公開鍵認証は秘密鍵という「本人しか持っていない、桁違いに長い鍵」を要求するため、総当たりで言い当てることが事実上不可能になります。この違いをただ文章で説明するのではなく、同じ攻撃ボタンを同じように押しても結果がまったく変わる、という体験としてユーザーに渡しているのがこの機能の狙いです。「対策をすると具体的にどれくらい安全になるのか」を、確率という数字の変化で直感的に掴んでもらう設計です。

実装面では、`triggerAttack`の第4引数`authMethod`を**任意(オプション)**にし、既定値を`'password'`にしています。これにより、この引数を渡していない既存の呼び出し箇所(もしあれば)は今までどおり動き続けます。関数のシグネチャを広げるときに、新しい引数を必須にせず任意にして安全な既定値を与える、というのは後方互換性を保つ典型的なテクニックです。

### 8-4: DDoSのボットネット風表現 — なぜ「描画オーバーレイ」として実装したのか

DDoS攻撃は本来、多数の乗っ取られた機器(ボット)が一斉に標的へトラフィックを送りつける攻撃です。この「多数のボットから同時に攻撃が来る」様子を表現したいところですが、素直に「ボットをネットワークのノードとして追加する」実装は選びませんでした。理由は第4章で説明した`useNetworkLayout.ts`の設計にあります。

このフックは「`nodes.length`(ノードの個数)、`width`/`height`が変化した時だけ`forceSimulation`を再構築する」という工夫でパフォーマンスを保っていました。つまり、DDoS攻撃のたびに一時的なボットノードを本物のノード配列に追加したり消したりすると、そのたびに`nodes.length`が変わり、d3-forceのシミュレーションが**再構築**されてしまいます。再構築が起きると、力学レイアウトは一度「ゆらぎ」ながら再収束するため、DDoSとは無関係な既存11ノードの配置まで揺れて見えてしまいます。教育目的で「攻撃が起きている場所」に注目してほしいのに、画面全体がガタつくのは逆効果です。

そこで`src/components/NetworkMap/BotnetLayer.tsx`は、ストアにもトポロジにもレイアウト計算にも一切触れない、**純粋な描画専用レイヤー**として作られています。表示するボットの数・角度・半径・タイミングは、`attack.id`(攻撃1件ごとに発行される一意なID)を種にした決定的なハッシュ関数(`hashString`、djb2系アルゴリズム)から計算しており、乱数を毎フレーム引き直すわけではありません。`useMemo`で攻撃ごとに1回だけ計算することで、同じ攻撃の間はボットの配置が固定され、余計なちらつきも起きません。

ボットの表示・非表示は、`activeAttacks`にDDoSの`AttackState`が存在するかどうかだけで決まる**ephemeral(一時的)な**性質を持たせています。`NetworkMap.tsx`側では`ddosAttacks = activeAttacks.filter(a => a.type === 'ddos')`を`BotnetLayer`に渡すだけで、DDoS攻撃が`removeAttack`でストアから消えると、`AnimatePresence`(Framer Motion)がそれを検知して自動的にフェード+スケールアウトさせます。「本体データを増やさず、既存の状態(activeAttacks)を見て一時的な見た目だけを足す」というこのアプローチは、パフォーマンス上のクリティカルパス(d3-forceの再構築)に触れずに視覚表現を拡張したいときの、再利用できる考え方です。
