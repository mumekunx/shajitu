## 第0章: このサイトは何？

### DoS/DDoSとは一言で

**DoS**（Denial of Service）とは「サービスを使えなくする攻撃」のことです。
レストランに例えると、悪意ある集団が席を全部占領して、本当のお客さんが入れなくする嫌がらせに似ています。

**DDoS**（Distributed DoS）は「分散した」版です。1人ではなく、世界中に散らばった何万台ものコンピュータが一斉に攻撃に加わります。

### このサイトのゴール

「DoSという概念を頭で理解する」だけでなく、「実際にどんな事件が起きたか、どんなスケールだったか」を映像として感じてもらうことが目的です。

2002年のDNS攻撃から2022年のソウル梨泰院まで、5つの事件をアニメーションで可視化しています。そして最後の2つ（Google Mapsハックと社会的DoS三連発）は、「悪意がなくてもDoSは起きる」という驚きのメッセージを伝えます。

## 第1章: 単一HTMLファイルって何がうれしい？

### HTMLとCSS、JavaScriptの関係

Webページを作る材料は3つあります。

- **HTML**: ページの「骨格」。見出し、段落、画像の置き場所を決める
- **CSS**: ページの「見た目」。色・フォント・レイアウト・アニメーションを決める
- **JavaScript**: ページの「動き」。クリックに反応したり、図を動かしたりする

普通は3つのファイルに分けますが、CSSは `<style>...</style>` タグ内に、JavaScriptは `<script>...</script>` タグ内に書けば、**1つのHTMLファイルに全部詰め込める**のです。

### GitHub Pagesで公開できる理由

GitHubにはリポジトリのファイルをそのままWebサーバーとして公開する「GitHub Pages」機能があります。
リポジトリのルートに `index.html` があれば、`https://ユーザー名.github.io/リポジトリ名/` でそのまま世界に公開されます。

単一ファイルならファイルが1つあるだけでOKなので、デプロイ（公開作業）が非常にシンプルです。サーバーを立てる必要も、ビルドツールを動かす必要もありません。

## 第2章: ダークテーマとCSS変数

### 色を変数にするとは？

このサイトはダークテーマ（黒背景）を使っています。背景色 `#0a0a0f`（ほぼ黒）、テキスト `#e0e0e0`（薄いグレー）、アクセント緑 `#00ff88`、アクセント赤 `#ff4444` などが主要な色です。

これらの色は「CSS変数」として一か所で定義されています。

```css
:root {
  --bg:    #0a0a0f;
  --green: #00ff88;
  --red:   #ff4444;
  --text:  #e0e0e0;
}
```

`:root` は「ページ全体のルート」という意味です。ここに `--変数名: 値;` の形で書いておくと、ページのどこでも `var(--green)` という書き方で参照できます。

### 変数にする利点

もし変数を使わずに `#00ff88` をページ中に100回書いていたら、色を1色変えたいときに100か所を直さないといけません。変数なら `:root` の1行を変えるだけで全部反映されます。

また、事件カードはそれぞれ違うテーマカラーを持っています（赤・オレンジ・紫・緑・黄）。JavaScriptからカードに `style.setProperty('--accent-color', '#ff4444')` のように書き込むことで、CSS側は `var(--accent-color)` で参照するだけでよくなります。「データとデザインを分ける」考え方です。

## 第3章: Canvasアニメーションの基本

### Canvasとは

`<canvas>` タグは「JavaScriptで自由に絵を描けるキャンバス（画布）」です。HTMLのタグとしては空っぽの箱で、JavaScriptから「ここに円を描いて」「ここに線を引いて」と命令して初めて絵が現れます。

```html
<canvas id="hero-canvas"></canvas>
```

```javascript
const canvas = document.getElementById('hero-canvas');
const ctx = canvas.getContext('2d'); // 2D描画モード
ctx.beginPath();
ctx.arc(100, 100, 20, 0, Math.PI * 2); // 座標(100,100)に半径20の円
ctx.fillStyle = '#00ff88';
ctx.fill();
```

### requestAnimationFrameで「パラパラ漫画」を作る

アニメーションの仕組みはシンプルです。

1. 絵を全部消す
2. 少し動かした位置に絵を描く
3. 1に戻る（繰り返す）

これを「1秒間に60回」行うのが `requestAnimationFrame`（略してrAF）という機能です。

```javascript
function draw() {
  ctx.clearRect(0, 0, W, H); // キャンバスをクリア
  // ここで絵を描く
  requestAnimationFrame(draw); // 次フレームでもdrawを呼ぶ
}
draw(); // 最初の1回を呼び出す
```

`requestAnimationFrame` は「次の画面更新タイミングで `draw` を呼んでください」とブラウザにお願いする関数です。ブラウザが自動でタイミングを調整してくれるため、なめらかなアニメーションが実現します。

### devicePixelRatioでくっきりさせる

スマートフォンや高解像度（Retina）ディスプレイでは、1ピクセルに2倍・3倍の物理的なドットが詰まっています。Canvasを普通に作ると「ぼやけた」見た目になります。

これを防ぐのが `devicePixelRatio`（dpr）への対応です。

```javascript
const dpr = window.devicePixelRatio || 1; // 通常は1, Retinaなら2
canvas.width  = 表示幅 * dpr;  // 実際のピクセル数を増やす
canvas.height = 表示高 * dpr;
ctx.scale(dpr, dpr);           // 描画座標は変えずに済む
```

こうすると、高解像度ディスプレイでもくっきりしたCanvasが表示されます。ヒーロー背景もすべての事件アニメーションも、この処理を行っています。

### ヒーロー背景を例に

ヒーロー背景のパーティクル（光点）は1つ1つが以下のデータを持ちます。

```javascript
{
  x: 横位置,
  y: 縦位置,
  vx: 横方向の速度,
  vy: 縦方向の速度,
  color: '#00ff88' または '#ff4444',
  alpha: 透明度,
  len: 軌跡の長さ,
}
```

毎フレーム、`p.x += p.vx; p.y += p.vy;` で位置を更新して再描画するだけでパーティクルが流れるように見えます。画面外に出たら新しいパーティクルに差し替えます。全体の密度は「画面面積 ÷ 8000」個に保たれます。

## 第4章: データとUIを分ける

### 事件データをJS配列に持つ

5つの事件カードはすべてJavaScriptのループで生成されています。事件のデータは `incidents` という配列に入っています。

```javascript
const incidents = [
  {
    id: 'dns',
    year: '2002 / 2007',
    title: 'DNSルートサーバー攻撃',
    summary: 'インターネットの「電話帳」13台が標的に...',
    accent: '#ff4444',
    details: ['事実1', '事実2', ...],
    // ...
  },
  // 残り4件
];
```

### ループでカードを作る発想

```javascript
incidents.forEach((inc, idx) => {
  // incの内容を使ってHTMLの要素を生成する
  const card = document.createElement('div');
  card.innerHTML = `<h3>${inc.title}</h3><p>${inc.summary}</p>`;
  container.appendChild(card);
});
```

`forEach` は配列の要素を1つずつ取り出してループする関数です。`inc` に現在の事件データが入り、それを使ってカードを作ります。

この設計の大きな利点は**「データを増やせばUIが増える」**ことです。事件を6件目に追加したければ、`incidents` 配列にオブジェクトを1つ追加するだけでカードが自動生成されます。HTMLを手で増やす必要はありません。

## 第5章: アコーディオン

### クリックで開閉する仕組み

「アコーディオン」とは、クリックすると折りたたまれていた内容が開き、もう一度クリックすると閉じるUIの名前です（楽器のアコーディオンが開閉するのと似ているから）。

仕組みはシンプルです。

```javascript
function toggleCard() {
  const expanded = card.classList.toggle('expanded');
  // expanded が true → 開いた状態
  // expanded が false → 閉じた状態
}
card.addEventListener('click', toggleCard);
```

`classList.toggle('expanded')` は、「`expanded` というクラスがなければ追加し、あれば削除する」という便利な関数です。CSSでは `.expanded .card-body { display: block; }` のようにして「expandedクラスがある時だけ中身を表示」と設定しておきます。

### 開いた時だけアニメを動かして軽くする工夫

Canvasアニメーションは毎フレーム計算を行うため、5つ全部を常に動かし続けると処理が重くなります。そこで「開いた時にアニメーションを開始し、閉じた時に止める」という設計にしています。

```javascript
if (expanded) {
  // 展開時: アニメーション開始
  window[`init_${inc.id}`]();
} else {
  // 折りたたみ時: rAFをキャンセルしてアニメーション停止
  window[`stop_${inc.id}`]();
}
```

`window.stop_dns()` などの関数の中では `cancelAnimationFrame(rafId)` を呼んでいます。これで次フレームの `draw` 呼び出しがキャンセルされ、CPUの無駄な計算が止まります。

### キーボード操作対応

クリックだけでなくキーボード（TabキーでフォーカスしてEnterキーで開閉）も対応しています。

```javascript
card.setAttribute('tabindex', '0');    // Tabキーでフォーカス可能に
card.setAttribute('role', 'button');   // スクリーンリーダーに「ボタン」と伝える
card.addEventListener('keydown', (e) => {
  if (e.key === 'Enter' || e.key === ' ') toggleCard();
});
```

これを「アクセシビリティ対応」と言います。マウスを使えない人でも操作できるようにする配慮です。

## 第6章: 5つの事件アニメの読み解き

### 事件1: DNSルートサーバー攻撃

**何を視覚化しているか**: インターネットのDNS（ドメイン名サービス）を担う13台のルートサーバーが攻撃される様子。

**攻撃の「殺到」をどう表現したか**: 13台のサーバーを円状に配置し、画面の四方（上下左右）からランダムにパケット（赤い点）が飛んでくるようにしました。パケットがサーバーに到達するたびにサーバーの「ヘルス（health）」が減り、0.3以下になると「DOWN」と点滅します。画面下部のゲージが「インターネット全体への影響」を示します。

### 事件2: エストニアサイバー攻撃

**何を視覚化しているか**: ロシア方向からエストニアの政府・銀行・メディアサイトへの22日間にわたる攻撃。

**攻撃の「殺到」をどう表現したか**: エストニアの国の形（多角形）を地図として描き、右側（ロシア方向）から60%の確率でパケットが出現します。攻撃12日目（5/9 ロシアの対ドイツ戦勝記念日）前後でスパイクが起き、パケット生成頻度が上がります。政府・銀行A・銀行B・メディアの4つのアイコンが順番にDOWNします。下部のタイムバーで22日間の進行が分かります。

### 事件3: Mirai Botnet

**何を視覚化しているか**: IoT機器（カメラ・ルーター・スマート電球）が感染してボット化し、Dynサーバーを攻撃する様子。

**攻撃の「殺到」をどう表現したか**: 60個のデバイスをランダムに散らばらせ、時間が経つにつれて緑（正常）から紫（感染）に変わります。感染数が50を超えると「攻撃フェーズ」に入り、感染デバイスから中央のDynサーバーへ一斉にパケットが殺到します。感染台数カウンターは最終的に150,000台超を示します（アニメ内の60台を比率で換算）。

### 事件4: Google Mapsハック（Simon Weckert）

**何を視覚化しているか**: ベルリン市街の道路グリッドを荷台車（と99台のスマホ）が歩き回り、Googleマップ上に渋滞が出現する様子。

**攻撃の「殺到」をどう表現したか**: 7×5のグリッドで道路を表し、荷台車（緑の小さな人形＋荷台）が通過した交差点の「congestion（混雑度）」が増加します。道路の色は緑（空いている）→黄（混んでいる）→赤（渋滞）と変化します。青い小さな車が渋滞エリアを検出すると迂回（オレンジに変わり別の道へ）します。これは「情報的DoS」——データを歪めることで機能を使えなくする——の視覚化です。

### 事件5: 社会的DoS三連発

**何を視覚化しているか**: 悪意なき「リクエスト集中」が歩道橋・電話網・会場をオーバーロードさせる様子を3つのタブで表現。

**攻撃の「殺到」をどう表現したか**: 3タブ（明石花火大会・電話輻輳・梨泰院）で同じアニメーションロジックを共用しています。流入アイコン（▲人 / ☎電話 / ●人）が三方から中央のサービス（歩道橋・電話網・会場）に向かって押し寄せます。上部のキャパシティバーが100%になると「OVERLOAD（過負荷）」状態に変わり、サービスが点滅してダウンします。タブを切り替えるとアニメがリセットされ、各事件を個別に観察できます。

## コラム: 「DoSは悪意だけじゃない」

このサイトで紹介した事件5件のうち、最後の2件（Google Mapsハックと社会的DoS三連発）には「攻撃者」がいません。

- Simon Weckertはアーティストとしての表現活動として荷台車を引いた
- 明石花火大会の観客は楽しみに集まった
- 震災後に安否確認の電話をかけた全国民は善意で行動した
- ハロウィンに集まったソウルの人々に悪意はなかった

それでも結果として「サービス（ナビ・歩道橋・電話・会場）が機能を失った」のです。

これらが示すのは「DoSとはリクエストの集中という構造的問題であり、悪意の有無とは関係ない」という事実です。

本物のサイバー攻撃（DNSルートサーバー攻撃・エストニア・Mirai）と社会的DoSに共通するのは、**「容量（キャパシティ）を超えるリクエストが集中すること」**という一点だけです。

このWebサイトがタイトルに「悪意ある攻撃から、リアカー1台まで」と書いているのはそのためです。プログラミングやネットワークを学ぶ際、「システムはどんなリクエストの集中に対しても脆弱になり得る」という視点を持っておくと、設計の考え方が大きく変わります。
