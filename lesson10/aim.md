# aim.md — ルーターファームウェア更新シミュレーター 実装の羅針盤

**このドキュメントの位置づけ**

aim.md は「なぜ作るか・どう進めるか」を実装着手前に確認するための羅針盤です。
各ドキュメントの役割分担は以下のとおりです。

| ファイル | 役割 |
|---|---|
| **aim.md**（このファイル） | 実装の狙いと進め方。着手前に読む |
| **detail.md** | 「何があるか」のリファレンス。ファイル構成・型・依存を記録 |
| **teach.md** | 「なぜ・どう動くか」の教科書。初学者向けの解説 |
| **update.md** | 進捗ログ。立案 → 進捗 → 完了の順に追記 |
| **tasks/todo.md** | 実装チェックリスト |

---

## 1. プロジェクトの目的（なぜ作るか）

### 何を作るか

架空のルーター **AS-3200HX** のファームウェア（ルーターの内部ソフトウェア）を更新する手順を、ブラウザ上でシミュレーションできる教材アプリです。

### なぜ作るか

ルーターのファームウェア更新は、セキュリティを保つうえで重要な作業です。しかし「実機（本物のルーター）を持っていない人」は、手順書を読んでも頭の中でイメージするしかありません。

このシミュレーターは、**実機なしでも作業の流れを体験できる**ことを目指します。

特に大切にしていることが1つあります。それは「**成功するだけでなく、つまずく・撤退する体験**」を盛り込むことです。

現実の作業では、うまくいかないことがよくあります。

- 管理画面にたどり着けない
- パスワードがわからない
- 途中で「やめたほうがいいかも」と判断する場面に直面する

こうした「つまずき」を安全な練習環境で先に体験しておくことで、初学者は「実際にやってみたら詰まった」という状況を想定しながら学べるようになります。

## 2. 成果物と技術制約

| 項目 | 内容 |
|---|---|
| 成果物 | `lesson10/index.html`（1ファイル） |
| 使用言語 | HTML / CSS / JavaScript（すべて同梱） |
| 外部ライブラリ | 使用しない（単一ファイルで完結） |
| 公開方法 | GitHub Pages にそのまま公開可能 |
| 架空設定 | 実在製品を模倣しない。型番 AS-3200HX の架空ルーター |
| ルーターの見た目 | シンプルなSVGイラストで描画 |

### なぜ単一ファイルなのか

単一ファイルにすることで、以下のメリットがあります。

- GitHub Pages でファイルを1つ置くだけで公開できる
- ビルドツールやサーバーが不要
- ファイルをコピーするだけで動く（配布が簡単）

## 3. 設計指針

シミュレーターを設計するうえで大切にする4つの指針です。

### ① リアルさ

実際のルーター管理画面に似た操作感（ログイン画面・設定メニューなど）を再現します。「これが本物の作業に近い」と感じてもらえる画面にすることで、学習効果が高まります。

### ② 分岐（成功ルートだけでなく、失敗・撤退ルートも）

成功するだけのシミュレーターは「正解の手順の暗記」にしかなりません。このシミュレーターでは、途中で迷ったり判断を迫られたりする分岐を用意します。失敗や撤退を選んだとしても、それが「正しい判断」になる場面を設けます。

### ③ 初学者向け

専門用語が出てきたら、その場に補足説明を表示します。「ファームウェア」「IPアドレス」「管理画面」など、当たり前のように使われる言葉をそのまま流さない設計にします。

### ④ 学習ポイントの明示

つまずいた箇所で「なぜここでつまずくのか」「実際の作業ではどう対処するのか」を画面上に出します。ゲームで言う「ヒント」のような役割です。つまずきが「学びのスポット」になるように設計します。

## 4. 実装 Phase

実装は段階的に進めます。各 Phase のゴールと主な作業は以下のとおりです。

| Phase | ゴール | 主な作業 |
|---|---|---|
| **Phase 0** | 準備完了 | 新規ブランチ作成・update.md に立案追記・todo.md にチェックリスト作成 |
| **Phase 1** | 骨組み完成 | 画面遷移の仕組み・状態管理の土台・スタート画面・ステップ進行の基盤を作る |
| **Phase 2** | ステップ1 実装 | ルーター本体確認画面・SVGイラスト・底面ラベル（型番 AS-3200HX・管理画面URL）の表示 |
| **Phase 3** | 分岐シナリオ本体 | ステップ2〜5のフロー・分岐A（初期PW警告）・分岐B（通信エラー）・分岐C（撤退確認）の実装 |
| **Phase 4** | エンディング・更新プロセス実装 | ステップ6（プログレスバー）・分岐D（文鎮化含む4エンディング）・操作ログエリアの実装 |
| **Phase 5** | 仕上げ | スタイル調整・レスポンシブ対応・文言の推敲 |
| **Phase 6** | ドキュメント整備・公開確認 | detail.md / teach.md 追記・動作確認・GitHub Pages 公開確認 |

### 各 Phase の詳細

**Phase 0 — 準備**
- `git status` でワークツリーがクリーンか確認
- ブランチ `feature/YYYYMMDD-HHMM-firmware-update-sim` を作成（日時は作業時点）
- update.md に立案エントリを追記
- tasks/todo.md にチェックリストを作成

**Phase 1 — 骨組み**
- 画面遷移の設計：どの「ステップ」にいるかを管理する状態（state）の仕組みを作る
- スタート画面：タイトル・架空ルーターのSVGイラスト・「開始」ボタン
- `showScreen(step)` のような関数で、画面を切り替える土台を整える

**Phase 2 — ステップ1 実装**
- ルーター本体確認の画面：「まず手元のルーターを確認しましょう」
- SVGイラストでルーターを描画（正面と底面）
- 底面ラベルに型番 **AS-3200HX**・管理画面URL（`http://192.168.1.1`）・初期パスワード（`admin`）を薄く表示

**Phase 3 — 分岐シナリオ本体**
- ステップ2（ログイン）: ID/PW入力・「初期設定のまま進める」ボタン
  - 分岐A: 初期パスワードのままログイン → 赤い警告バナー「⚠️ 初期パスワードのまま使用されています」を以降すべての画面に表示（Mirai的IoT弱PW体験）
- ステップ3（管理トップ）: サイドバー（インターネット/LAN/無線設定/ファームウェア更新）・現バージョン「Ver 1.00(2021年リリース)」・「新しいVer 1.11が利用可能」通知バッジ
- ステップ4（更新画面）: 「ローカルファイル指定」「オンラインバージョンアップ」の2択
  - 分岐B: オンライン → プログレス50%で「通信エラー:サーバーに接続できません」→「もう一度試す」or「ローカルファイル指定に切り替える」
  - ローカル: 「ファイルを選択」（ダミー）→ 架空ファイル名「wsr_xxxx_jp_111.bin」自動入力 → 「更新実行」
- ステップ5（更新確認）: 警告「本当に更新しますか？更新中は電源を切らないでください」
  - 分岐C: 「実行する」→ 更新プロセスへ / 「キャンセル」→ 撤退エンディングへ
- 学習ポイント（解説テキスト）の表示ロジック

**Phase 4 — エンディング・更新プロセス実装**
- ステップ6（更新プロセス）: プログレスバー（CSSアニメ）・「更新中... 絶対に電源を切らないでください」注意書き
  - 分岐D（ユーザーにボタンで選ばせる方式）: 通常完了 → 100% → 成功エンディング / 中断トラブル → 80%で停止「⚠️ 通信が切断されました」→「再起動を試す」→「復旧」or「文鎮化」にユーザー選択で分岐
- エンディング4種を実装:
  1. **完全成功**: 「ファームウェア更新が完了しました(Ver 1.11)」/ PW変更済みなら警告バナー消える
  2. **撤退**: 「更新を中止しました」/ 撤退も立派な判断というメッセージ
  3. **通信トラブルからの復旧**: 「再起動後、正常に動作していることを確認できました」
  4. **文鎮化（最悪）**: 「ルーターが応答しなくなりました」/ 画面グレーアウト演出
- 各エンディング共通: 「もう一度シミュレーションする」ボタン・操作ログの全ログ振り返り
- 操作ログエリア: ユーザーの選択を時系列テキストログで蓄積（等幅フォントで表示）

**Phase 5 — 仕上げ**
- 全体のスタイル（フォント・色・余白）を整える
- スマートフォンでも使えるレスポンシブ対応
- 画面テキストの言葉遣いを確認・推敲

**Phase 6 — ドキュメント整備・公開確認**
- detail.md にファイル構成・型・依存を追記
- teach.md に初学者向け解説章を追記
- ブラウザ・スマートフォンで動作確認
- GitHub Pages で公開され正しく表示されるか確認

## 5. ドキュメント運用ルール（CLAUDE.md 準拠）

### ファイル別の役割と更新タイミング

| ファイル | 更新タイミング |
|---|---|
| **update.md** | 依頼を受けたとき（立案）→ 作業中（進捗）→ 完了時（完了マーク）|
| **detail.md** | 新規ファイル作成時・既存ファイルを大幅変更したとき・ファイル削除・リネーム時 |
| **teach.md** | 新機能・重要パターンを追加したとき（細かい変更では毎回更新しない） |
| **tasks/todo.md** | Phase 0 で作成。各 Phase の着手前・完了時にチェックを更新 |

### ブランチ運用

- ブランチ名: `feature/YYYYMMDD-HHMM-firmware-update-sim`
  - 例: `feature/20260615-1030-firmware-update-sim`
  - 日時は `date +%Y%m%d-%H%M` で取得する
- main ブランチには直接コミットしない
- マージはユーザーが PR レビュー後に行う（Claude は merge/push を勝手にしない）

### サブエージェント運用

プロジェクト成果物ファイル（index.html・detail.md・teach.md・update.md など）の作成・編集は、**Sonnet サブエージェント経由**で行います。メイン（Opus）は計画・調査・指示・結果確認のみを担当します。
